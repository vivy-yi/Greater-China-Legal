# 软件著作权侵权分析

## 工作流程

```
1. 读取 ../../CLAUDE.md → 获取管辖地/数据源配置/置信度规则
2. 提取案件基本信息 → 软件类型/权利代码特征/涉嫌侵权行为
3. 构建权利基础 → 软件著作权登记情况/开源许可状态
4. 侵权分析判断树 → 执行实质性相似判断
5. 开源合规风险评估 → GPL/Apache/MIT等许可证传染性分析
6. 最终用户侵权认定 → 授权范围/使用场景分析
7. 证据充分性评估 → 现有证据+补充建议
8. 输出分析结论与建议
```

## 一、适用情形

| 场景 | 典型描述 | 分析重点 |
|------|----------|----------|
| 代码抄袭投诉 | "对方软件界面/代码与我方相似" | 实质性相似+接触可能性 |
| 开源合规争议 | "我方软件是否受GPL许可证传染" | GPL传染性+授权链完整性 |
| 最终用户侵权 | "对方未经授权复制使用我方软件" | 授权范围+商业使用认定 |
| 软件著作权登记异议 | "对方登记的软件与我方代码重复" | 权利归属+在先权利比对 |
| 离职员工代码归属 | "前员工带走了代码用于新公司" | 职务作品认定+竞业限制 |
| 委托开发成果争议 | "委托开发的软件版权归谁" | 合同约定+著作权法规则 |

## 二、权利基础构建

### 2.1 软件著作权登记审查

| 信息项 | 审查内容 | 风险点 |
|--------|----------|--------|
| 登记证书 | 证书编号/软件名称/著作权人 | 未登记则需其他证据佐证 |
| 登记时间 | 首次发表时间/登记时间 | 推算保护期（50年） |
| 权利范围 | 整件软件还是部分模块 | 部分登记仅保护登记部分 |
| 变更/转让 | 是否有转让或授权记录 | 授权链条是否完整 |

### 2.2 权利归属判断

```
软件著作权归属判断树：

├── 是否为职务软件？
│   ├── 执行本单位工作任务 → 单位为著作权人（合同另有约定除外）
│   │   ├── 主要利用本单位物质技术条件 → 单位享有著作权（约定优先）
│   │   └── 兼职/临时员工 → 需审查雇佣关系实质
│   └── 委托开发软件 → 合同约定著作权归属；无约定则受托方享有
│       └── 须审查委托合同条款
│
├── 是否为合作开发软件？
│   ├── 两个以上单位/个人合作开发
│   │   ├── 有约定按约定
│   │   └── 无约定：各方共同享有，按投入比例分配（可分割使用的可分割部分各自享有）
│
└── 继受取得（转让/继承）？
    └── 须核查转让合同/继承证明/授权许可协议
```

### 2.3 保护范围认定

**软件著作权保护什么：**

- ✅ 受保护：源代码（源程序）、目标代码、程序说明书、界面设计（独创性表达）
- ❌ 不受保护：算法思想、操作方法、技术方案、通用命令/术语、效率优化（属于思想而非表达）

## 三、侵权分析判断树

### 3.1 总体判断框架

```
                    ┌─────────────────────────────┐
                    │  第一步：权利基础审查        │
                    │  是否有有效软件著作权？      │
                    └──────────┬──────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  第二步：接触可能性  │
                    │  被告是否可能接触原告│
                    │  软件（源代码/文档） │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────────────┐
                    │  第三步：实质性相似判断      │
                    │  两软件是否构成实质性相似？  │
                    └──────────┬──────────────────┘
                               │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        【是：高度相似】 【否：差异明显】 【不确定：需鉴定】
              │               │               │
              ▼               ▼               ▼
        继续第四步       不构成侵权      申请法院鉴定
                                        或技术鉴定机构
              │
    ┌─────────┴───────────────┐
    ▼                       ▼
【第四步：创意过滤】   【第五步：开源合规】
  剔除思想/通用表达      审查授权链与许可合规
    │                       │
    ▼                       ▼
  过滤后仍相似           侵权认定+赔偿计算
  → 构成侵权            或→不构成侵权
```

---

### 3.2 实质性相似判断（核心步骤）

#### 3.2.1 代码比对方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| 整体观感法 | 从整体上比对两软件的结构、流程、界面 | 界面相似、整体抄袭 |
| 逐行比对法 | 源代码逐行/逐函数比对 | 代码高度相似、修改后仍相似 |
| 抽象测试法 | 将代码逐层抽象（具体代码→流程→概念），忽略不受保护的思想 | 复杂软件、架构相似 |
| 过滤测试法 | 剔除不受保护元素（思想/通用表达/公知技术）后比对 | 涉及算法/效率优化的争议 |
| 第三方工具 | 使用代码相似度检测工具（如Simian/JPlag） | 大规模代码库比对 |

#### 3.2.2 实质性相似判断树

```
实质性相似判断：

Step 1：确定比对对象
├── 原告主张权利的代码范围
│   ├── 全部软件还是特定模块
│   └── 是否有选择性地仅登记部分代码
│
└── 被告软件的哪些部分
    ├── 源代码（最直接证据）
    ├── 目标代码（逆向工程获取）
    └── 软件界面/架构（间接证据）

Step 2：初步比对
├── 是否存在相同的错误代码（死代码/注释）
├── 变量/函数命名是否相同
├── 代码结构和流程是否高度一致
└── 是否使用相同的独创性算法

Step 3：过滤不受保护内容
├── 思想层面的相似（功能/流程相似不算侵权）
│   └── 如：都用"快速排序"算法 → 不构成侵权
│
├── 公有领域的表达（通用编程方法）
│   └── 如：标准的数据库CRUD操作 → 不构成侵权
│
└── 有限的表达（效率优化、唯一表达方式）
    └── 如：只有一种方式实现该功能 → 不构成侵权

Step 4：综合判断
├── 过滤后仍有实质性相似 → 构成侵权
├── 仅思想相似但表达不同 → 不构成侵权
└── 相似部分属于公有领域 → 不构成侵权
```

#### 3.2.3 接触可能性判断

| 证据类型 | 说明 | 证明力 |
|----------|------|--------|
| 源代码泄露 | 被告曾接触过原告源代码（劳动合同/项目合同/ NDA） | 高 |
| 软件演示 | 被告曾参与原告软件演示/测试 | 中 |
| 公开可得 | 原告软件曾公开发布（网站/应用商店/论文） | 中 |
| 离职员工 | 被告公司有原告前员工 | 需结合其他证据 |
| 委托关系 | 原告曾委托被告开发/维护 | 高 |

---

### 3.3 开源许可证合规分析

#### 3.3.1 主要开源许可证分类

| 许可证 | 传染性 | 核心义务 | 典型例子 |
|--------|--------|----------|----------|
| **GPL v2/v3** | 强传染（Copyleft） | 修改/衍生作品须以GPL开源 | Linux内核/GCC |
| **LGPL** | 弱传染 | 链接库可闭源，但修改库本身须开源 | glibc |
| **Apache 2.0** | 无传染 | 须保留许可证声明和修改说明 | Hadoop/Maven |
| **MIT** | 无传染 | 须保留许可证声明 | jQuery/React（部分） |
| **BSD** | 无传染 | 须保留许可证声明 | Node.js（部分） |
| **MPL** | 中等传染 | 修改文件须开源，整体可闭源 | Firefox |
| **AGPL** | 强传染+网络 | 网络使用也须开源 | MongoDB |

#### 3.3.2 GPL传染性判断树

```
GPL传染性分析：

问题1：软件是否包含GPL代码？
├── 是 → 整个软件须以GPL开源（除非符合LGPL链接豁免）
└── 否 → 继续下一步

问题2：是否通过动态/静态链接使用GPL库？
├── 动态链接 → 通常触发LGPL豁免，可闭源（但修改库本身须开源）
├── 静态链接 → 一般触发GPL传染，整个软件须开源
└── ⚠️ 警惕：即使"仅调用API"，若深度集成仍可能构成"衍生作品"

问题3：是否通过API调用远程服务（网络聚合）？
├── 是 + 使用AGPL → 网络使用也须开源（AGPL特别条款）
└── 是 + GPL → 须审查是否构成"通过网络交互"触发传染

问题4：是否有有效的许可证授权链？
├── 引入开源组件时是否保留原许可证声明
├── 是否向用户提供完整的许可证文本
└── ⚠️ 许可证冲突（如同时使用GPL和闭源专有组件）→ 须隔离或协商

结论输出：
├─ 高风险：GPL传染成立 → 须开源/删除代码/寻求商业许可
├─ 中风险：需进一步分析链接方式和集成深度
└─ 低风险：许可证无传染性或已合规
```

#### 3.3.3 开源合规风险矩阵

| 风险等级 | 场景 | 后果 | 建议 |
|----------|------|------|------|
| 🔴 高 | GPL代码直接嵌入商业软件并发布 | 须开源+可能承担侵权责任 | 立即整改，删除/替换GPL代码 |
| 🟠 中 | 动态链接GPL库但未保留声明 | 违反许可证，可能被追诉 | 补充声明，评估替代方案 |
| 🟡 低 | Apache/MIT/BSD许可证合规 | 合规使用无传染风险 | 保留声明即可 |
| 🟢 无 | 完全自主开发，无开源组件 | 无传染风险 | 保持自主代码审计 |

### 3.4 最终用户侵权认定

#### 3.4.1 侵权行为类型

| 行为类型 | 定义 | 侵权构成要件 |
|----------|------|-------------|
| 复制权 | 将软件安装到计算机等设备 | 未经授权复制安装 |
| 修改权 | 改编、翻译软件 | 未经授权修改 |
| 信息网络传播权 | 通过网络向公众提供 | 上传至网络供下载 |
| 商业使用 | 用于商业经营活动 | 以营利为目的使用 |
| 出租权 | 出租软件原件/复制件 | 商业性出租 |

#### 3.4.2 最终用户侵权判断树

```
最终用户侵权分析：

问题1：软件类型？
├── 通用软件（操作系统/办公软件等）
│   ├── 企业/商业使用 → 通常需要商业授权
│   └── 个人/教育使用 → 需审查最终用户许可协议（EULA）
│
└── 定制软件/嵌入式软件
    ├── 随硬件一并销售 → 须审查许可协议是否允许
    └── 单机/网络使用 → 须判断授权范围

问题2：授权许可类型？
├── 单机授权 → 只能在指定设备使用
├── 企业授权 → 可在企业内网多用户使用
├── 源代码授权 → 可查看/修改/编译部署
└── ⚠️ 严格审查"不得逆向工程"条款

问题3：使用目的是否属于合理使用？
├── 课堂教学/科学研究 → 可能适用合理使用（须严格限定）
├── 个人学习研究 → 通常仅限学习目的
└── 商业生产环境 → 须商业授权，否则构成侵权

问题4：是否存在软件盗版？
├── 使用破解补丁/注册机 → 直接侵权
├── 超过授权数量的安装 → 超出部分侵权
└── 使用过期许可证 → 未经授权继续使用构成侵权

问题5：是否为职务使用？
├── 单位员工在工作电脑上安装 → 单位可能承担连带责任
└── ⚠️ 企业有义务建立软件合规管理制度
```

## 四、具体分析步骤

### 第一阶段：信息提取与权利基础确认

**步骤1.1：收集权利信息**

```
收集以下信息（如用户未提供则标注"待补充"）：
□ 软件名称及版本
□ 软件著作权登记证书编号
□ 登记时间及首次发表时间
□ 权利范围（全部/部分模块）
□ 是否有转让或授权记录
□ 相关源代码文档（如有）
□ 开源组件使用情况清单
```

**步骤1.2：收集涉嫌侵权信息**

```
□ 涉嫌侵权软件名称/版本
□ 涉嫌侵权方信息（公司名/个人）
□ 发现渠道（投诉/市场监测/举报）
□ 涉嫌侵权行为类型（复制/修改/传播）
□ 涉嫌侵权的具体表现（代码相似/界面相似/功能复制）
□ 发现时间/持续时间
□ 已有证据清单
```

**步骤1.3：建立事实对照表**

| 信息项 | 权利软件 | 涉嫌侵权软件 | 一致性评估 |
|--------|-----------|--------------|------------|
| 软件类型 | | | |
| 核心功能 | | | |
| 界面设计 | | | | |
| 代码结构 | | | |
| 关键算法 | | | |
| 错误特征 | | | |
| 文档风格 | | | |

---

### 第二阶段：实质性相似技术分析

**步骤2.1：代码层面比对**

```
□ 提取原告主张的核心代码段（通常为独创性高的部分）
□ 获取被告软件的可比对代码（源代码或目标代码逆向结果）
□ 进行逐行比对，记录：
  - 完全相同的代码行
  - 高度相似的代码段（变量名/函数名不同但结构相同）
  - 相同的注释内容或错误特征
  - 相同的命名规范/编码风格
□ 统计相似代码行数占总代码的比例
□ ⚠️ 注意：比例不是唯一标准，独创性核心部分相似即可认定
```

**步骤2.2：抽象与过滤**

```
□ 对比对结果进行抽象层分析：
  - L1（具体代码层）：具体源代码行
  - L2（结构层）：函数划分、流程控制结构
  - L3（架构层）：模块划分、数据流设计
  - L4（概念层）：软件要解决的核心问题/创意

□ 对每一层进行过滤：
  - L4概念层（思想）→ 不受保护，过滤
  - L3架构层（通用设计模式）→ 不受保护，过滤
  - L2结构层（有限表达）→ 需具体分析
  - L1代码层（具体表达）→ 受保护

□ 过滤后分析：
  - 若L1层实质性相似 → 构成侵权
  - 若L2层存在有限表达且相似 → 可能构成侵权
  - 若仅L3/L4层相似 → 不构成侵权
```

**步骤2.3：接触可能性评估**

```
□ 评估被告接触原告软件的可能性：
□ 证据线索：
  - 商业合作关系（是否有过合作/接触）
  - 雇佣关系（是否有原告前员工）
  - 合同关系（是否签过NDA/保密协议）
  - 公开渠道（原告软件是否公开发布）
  - 委托关系（是否曾委托被告开发）
□ 接触证据强度评分：高/中/低/无
```

---

### 第三阶段：开源合规专项分析

**步骤3.1：开源组件清单审查**

```
□ 要求提供涉嫌侵权软件的全部开源组件清单
□ 逐一核查每个组件的许可证类型
□ 识别GPL/AGPL/LGPL等传染性许可证的使用情况
□ 评估许可证之间的兼容性
□ ⚠️ 注意：商业软件中常混入开源组件，须全面排查
```

**步骤3.2：许可证合规性审查**

| 许可证 | 必须保留的声明 | 传染性要求 | 合规要点 |
|--------|---------------|------------|----------|
| GPL v2/v3 | 版权声明+GPL全文 | 衍生作品须以GPL开源 | 整个软件开源或删除 |
| LGPL | 版权声明+LGPL全文 | 修改库须开源，链接可闭源 | 隔离或获取商业许可 |
| Apache | 版权声明+许可文本 | 修改须说明 | 保留声明+修改说明 |
| MIT | 版权声明 | 无传染 | 保留声明即可 |
| BSD | 版权声明 | 无传染 | 保留声明 |

**步骤3.3：传染风险量化评估**

```
评估公式：高传染组件数量 × 使用方式风险系数 × 分发范围系数

风险系数：
├─ 直接嵌入源代码（高）
├─ 静态链接编译（高）
├─ 动态链接运行时调用（中）
└─ 仅参考设计思想（低）

分发范围：
├─ 对外公开发布（高）
├─ 仅企业内部使用（中）
└─ 未实际分发（低）
```

### 第五阶段：综合结论与维权建议

**步骤5.1：侵权认定综合判断**

```
┌─────────────────────────────────────────────────────────┐
│                   综合判断标准                          │
├─────────────────────────────────────────────────────────┤
│ 构成著作权侵权须同时满足：                               │
│  ① 原告对涉案软件享有有效的著作权                       │
│  ② 被告有机会接触到原告软件                             │
│  ③ 被告软件与原告软件构成实质性相似                     │
│  ④ 相似部分属于原告受保护的独创性表达                   │
└─────────────────────────────────────────────────────────┘

判断结论选项：
├─ 🟢 不构成侵权：权利基础不存在 / 无接触可能 / 无实质性相似 / 相似部分不受保护
├─ 🟡 可能构成侵权（需进一步鉴定）：存在接触+部分相似，但无法确定独创性程度
└─ 🔴 构成侵权：权利基础有效+接触可能性+实质性相似
```

**步骤5.2：赔偿金额估算**

| 侵权类型 | 计算方式 | 优先顺序 |
|----------|----------|----------|
| 实际损失 | 权利人实际损失（如销量下降） | 第一优先 |
| 侵权获利 | 侵权人因侵权所获利益（须举证） | 第二优先 |
| 参考定额 | 软件500元-500万元（综合考虑） | 无法计算前两者时 |
| 合理开支 | 律师费/公证费等（须合理） | 另行主张 |

**步骤5.3：维权路径选择**

| 路径 | 适用情形 | 优点 | 缺点 | 周期 |
|------|----------|------|------|------|
| 协商和解 | 侵权轻微、双方有意愿 | 快、成本低 | 无强制执行 | 1-4周 |
| 发送律师函 | 侵权明确、需要施压 | 中等成本、效果快 | 可能激化矛盾 | 1-2周 |
| 行政投诉 | 版权侵权（文化旅游执法） | 快、费用低 | 罚款力度有限 | 1-3月 |
| 民事诉讼 | 损失较大、需要赔偿 | 效果好、有强制力 | 周期长（6-18月） | 6-18月 |
| 刑事报案 | 严重侵权、达到立案标准 | 最强震慑 | 立案门槛高 | 不确定 |

**刑事立案标准（侵犯著作权罪）：**

| 情形 | 数额/数量标准 |
|------|---------------|
| 违法所得3万元以上 | 刑事责任 |
| 非法经营额5万元以上 | 刑事责任 |
| 复制发行计算机软件500份以上 | 刑事责任 |
| ⚠️ 单位犯罪的，标准按上述5倍执行 | |
