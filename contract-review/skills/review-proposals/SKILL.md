# /review-proposals — China Mainland 合同审查提案咨询

## Purpose / 目的

本技能用于逐步审查 playbook-monitor agent 待处理的 playbook 更新提案，并将经批准的变更安全地应用到 `../CLAUDE.md` 主配置文件中。

在商业合同审查场景中，本技能模拟法律顾问对合同条款变更提案进行系统性审查的完整流程，确保每项变更均经过充分评估、合理授权并可追溯记录。

---

## 适用范围

本技能适用于以下情形：

1. **主动触发**：用户询问"有哪些待处理提案"、"审查合同提案"、"待处理的合同变更有哪些"
2. **被动触发**：playbook-monitor agent 自动提交了新的合同审查规则或条款变更提案
3. **周期性审查**：定期检查并清理过期的待处理提案
4. **批量审查**：同时处理多个相关提案，需要整体评估其相互影响

---

## 核心判断树 / Decision Tree

### 第一层判断：提案文件状态

```
开始
  │
  ▼
提案文件是否存在？
  │
  ├── 否 → 输出：无待处理提案，Playbook 已是最新的
  │         结束
  │
  └── 是 → 提案文件是否为空？
            │
            ├── 是 → 输出：提案文件为空，无待处理内容
            │         结束
            │
            └── 否 → 进入第二层判断
                      ▼
```

### 第二层判断：提案类型识别

```
提案文件不为空
  │
  ▼
识别提案类型：
  │
  ├── 条款新增提案 → 评估是否与现有条款冲突
  │                 │
  │                 ├── 是 → 标记为【冲突提案】
  │                 └── 否 → 标记为【新增提案】
  │
  ├── 条款修改提案 → 评估修改幅度和影响范围
  │                 │
  │                 ├── 幅度≤10% → 标记为【微调提案】
  │                 └── 幅度>10% → 标记为【重大修改提案】
  │
  ├── 条款删除提案 → 评估删除的风险和必要性
  │                 │
  │                 └── 标记为【删除提案】
  │
  └── 混合提案 → 逐一分类后进入批量处理流程
```

### 第三层判断：审查决策

```
对每个提案进行审查
  │
  ▼
用户选择操作：
  │
  ├── 【接受】→ 显示精确 diff → 用户确认？
  │             │
  │             ├── 确认 → 应用变更 → 记录日志
  │             └── 拒绝 → 返回修改选项
  │
  ├── 【拒绝】→ 记录拒绝理由 → 不修改 CLAUDE.md
  │
  ├── 【修改】→ 进入条款修改协商流程
  │             │
  │             └── 重新生成提案 → 循环至用户确认
  │
  └── 【延期】→ 记录延期原因及预期处理时间
                保留提案文件供后续处理
```

### 第四层判断：批量处理终态

```
所有提案处理完毕？
  │
  ├── 否 → 返回继续处理下一个提案
  │
  └── 是 → 生成变更摘要 → 存档提案文件
                输出最终报告
                结束
```

## 详细操作步骤 / Specific Steps

### 步骤 1：环境准备与提案加载

**目标**：确认工作目录、加载 playbook-monitor agent 及待处理提案文件。

**执行内容**：

1. 确认当前工作目录为 `/tmp/gl-work/contract-review/`
2. 检查提案文件是否存在：`./proposals/pending-proposals.md`
3. 读取提案文件内容并解析
4. 统计待处理提案数量
5. 按类型对提案进行预分类

**输出中间状态**：
- 已加载提案文件
- 提案总数：N
- 分类统计：新增 M 条，修改 K 条，删除 L 条

**异常处理**：
- 文件不存在：输出标准结束语
- 文件为空：输出空文件提示
- 文件格式异常：尝试解析，失败则提示手动检查

---

### 步骤 2：逐个展示提案

**目标**：对每个待处理提案进行清晰展示，包含条款内容、变更说明、适用法律依据。

**执行内容**：

对每个提案显示以下信息块：

```
╔══════════════════════════════════════════════════════╗
║  提案编号：PP-2026-XXXX                               ║
║  提案类型：【新增/修改/删除】                          ║
║  提交时间：YYYY-MM-DD HH:MM                           ║
╠══════════════════════════════════════════════════════╣
║  条款位置：第 X 章 / 第 X 条                          ║
║  关联法规：民法典第 XXXX 条                           ║
╠══════════════════════════════════════════════════════╣
║  变更说明：                                           ║
║  [此处详细描述变更内容]                                ║
╠══════════════════════════════════════════════════════╣
║  变更前：                                             ║
║  [原条款内容，verbatim]                                ║
╠══════════════════════════════════════════════════════╣
║  变更后：                                             ║
║  [新条款内容，verbatim]                                ║
╠══════════════════════════════════════════════════════╣
║  风险评估：                                           ║
║  - 法律风险等级：[高/中/低]                            ║
║  - 商业影响：[有利/中性/不利]                          ║
║  - 实施复杂度：[简单/中等/复杂]                        ║
╚══════════════════════════════════════════════════════╝
```

**用户操作选项**：
- 输入 `A` 或 `接受`：接受此提案
- 输入 `R` 或 `拒绝`：拒绝此提案
- 输入 `M` 或 `修改`：进入修改协商流程
- 输入 `D` 或 `延期`：延期处理
- 输入 `S` 或 `跳过`：暂时跳过此提案，处理下一个
- 输入 `Q` 或 `退出`：终止审查流程

---

### 步骤 3：接受提案的处理流程

**前提**：用户选择了【接受】选项。

**执行内容**：

1. **生成精确 Diff**：使用统一格式展示变更差异
2. **显示变更预览**：将应用于 CLAUDE.md 的确切内容
3. **等待用户确认**：要求用户输入确认指令
4. **应用变更**：在用户明确确认后修改 CLAUDE.md
5. **记录日志**：将变更记录到变更日志文件

**Diff 输出模板**：

```
--- CLAUDE.md（变更前）
+++ CLAUDE.md（变更后）
@@ 第 X 行 @@
- 原内容
+ 新内容
```

**确认指令要求**：
- 用户必须输入：`确认` 或 `apply`（大小写不敏感）
- 输入其他内容视为拒绝，将返回提案展示界面

**日志记录内容**：
- 提案编号
- 变更类型
- 变更内容摘要
- 审批人
- 审批时间
- 变更后版本号

---

### 步骤 4：拒绝提案的处理流程

**前提**：用户选择了【拒绝】选项。

**执行内容**：

1. **记录拒绝理由**：要求用户选择或输入拒绝原因
2. **评估影响**：简要分析拒绝此提案的影响
3. **生成替代建议**：如适用，提供替代方案
4. **更新提案状态**：将提案标记为已拒绝
5. **不修改 CLAUDE.md**：确认不对主文件进行任何变更

**拒绝理由分类**（用户选择或输入）：
- `1`：与现行法规存在冲突
- `2`：与公司政策不符
- `3`：商业风险过高
- `4`：表述存在歧义或错误
- `5`：时机不成熟
- `6`：其他（需用户输入说明）

**拒绝日志模板**：

```
## 拒绝记录
- 提案编号：PP-2026-XXXX
- 拒绝时间：[时间戳]
- 拒绝理由：[选择的原因代码和/或用户输入说明]
- 拒绝操作人：[操作人标识]
- 替代建议：[如有]
```

---

### 步骤 5：修改提案的处理流程

**前提**：用户选择了【修改】选项。

**执行内容**：

1. **进入协商模式**：展示当前提案内容
2. **逐项确认修改点**：对每个可能修改的部分逐一征求用户意见
3. **生成修改建议**：基于用户反馈生成新版本提案
4. **预览修改结果**：展示修改后的完整条款
5. **用户确认**：用户确认修改后，将修改版本作为新提案重新进入审查流程
6. **循环处理**：新提案从步骤 2 重新开始

**修改协商流程**：

```
修改协商开始
  │
  ▼
展示当前提案全文
  │
  ▼
用户输入修改指令：
  │
  ├── 行号+新内容 → 仅修改指定行
  ├── 段落引用+新内容 → 替换整个段落
  ├── 插入位置+新内容 → 在指定位置插入
  ├── 删除行号/段落 → 删除指定内容
  └── 自由编辑 → 进入全文编辑模式
        │
        ▼
预览修改结果
  │
  ▼
用户确认？
  │
  ├── 确认 → 生成新提案 → 重新进入审查流程
  └── 继续修改 → 返回修改协商
```

---

### 步骤 6：延期提案的处理流程

**前提**：用户选择了【延期】选项。

**执行内容**：

1. **记录延期原因**：要求用户选择或输入延期原因
2. **设定预期处理时间**：用户指定或使用默认时间
3. **更新提案状态**：将提案标记为已延期
4. **保留提案文件**：延期期间保留提案供后续处理
5. **设置提醒**：如系统支持，在预期时间提醒处理

**延期原因分类**：
- `1`：等待相关法规明确
- `2`：等待内部审批流程完成
- `3`：等待外部当事人反馈
- `4`：需要进一步法律研究
- `5`：业务优先级调整
- `6`：其他（需用户输入说明）

**延期时间选项**：
- `1W`：延期 1 周
- `2W`：延期 2 周
- `1M`：延期 1 个月
- `3M`：延期 3 个月
- `TBD`：待定（需手动激活）

**延期记录模板**：

```
## 延期记录
- 提案编号：PP-2026-XXXX
- 延期时间：[时间戳]
- 延期原因：[选择的原因代码和/或用户输入说明]
- 预期处理时间：[设定的延期截止时间]
- 状态：延期处理中
```

### 步骤 7：批量处理与进度跟踪

**目标**：在存在多个待处理提案时，维护处理进度并提供清晰的整体视图。

**执行内容**：

1. **显示处理进度**：当前处理第 N 个，共 M 个
2. **显示待处理摘要**：剩余提案类型统计
3. **支持跳跃处理**：用户可选择跳到下一个或指定编号的提案
4. **暂停与恢复**：支持用户暂停并在后续恢复审查

**进度显示模板**：

```
╔══════════════════════════════════════════════════════╗

╠══════════════════════════════════════════════════════╣
║  总计：10 个提案                                      ║

║  ├─ 新增：2 个                                        ║
║  ├─ 修改：4 个                                        ║
║  └─ 删除：1 个                                        ║

║  当前进度：4 / 10                                     ║
║  当前提案：PP-2026-0004                              ║
╚══════════════════════════════════════════════════════╝
```

**批量操作命令**：
- `next`：处理下一个提案
- `prev`：返回上一个提案
- `jump N`：跳转到第 N 个提案
- `summary`：显示当前处理摘要
- `pause`：暂停批量处理
- `resume`：恢复批量处理

**前提**：所有提案均已处理完毕。

**执行内容**：

1. **生成变更摘要报告**：汇总所有变更内容
2. **统计变更数据**：新增、修改、删除数量统计
3. **更新版本号**：如适用，更新 CLAUDE.md 版本号
4. **存档提案文件**：将已处理的提案移至存档目录
5. **输出最终报告**：向用户展示完整的处理结果
