## 1. 三层配置

| 配置层 | 用途 | 数据结构 | 持久化路径 |
|---|---|---|---|
| 黑名单 | 强制脱敏，不受其他规则限制 | `{ original, type }[]` | `plugins/legal-scenes/<本场景>/matters/<slug>/redaction-blacklist.json` |
| 白名单 | 覆盖检查：包含白名单词的实体跳过 | `string[]` | `plugins/legal-scenes/<本场景>/matters/<slug>/redaction-whitelist.json` |
| 自定义类型 | 用户定义的新脱敏类型（合同名称/产品型号等） | `{ id, name, items: string[] }[]` | `plugins/legal-scenes/<本场景>/matters/<slug>/redaction-custom.json` |

## 2. 优先级顺序（高 → 低）

```
┌─────────────────────────────────────────────┐
│  1. 黑名单（最高）— 强制脱敏，不受白名单限制  │
│     ↓                                        │
│  2. 白名单 — 覆盖检查：包含白名单词的实体跳过   │
│     ↓                                        │
│  3. 自定义类型 — 用户定义的新类型              │
│     ↓                                        │
│  4. 内置 16 类规则（最低）— 默认正则匹配       │
└─────────────────────────────────────────────┘
```

**示例**：
- 黑名单有"AA"、白名单有"AABB" → "AA"仍被识别（黑名单优先）
- 白名单有"计划"、内置规则识别到"股权激励计划" → 跳过（白名单覆盖）

## 3. 黑名单（强制脱敏）

**用途**：内置规则未匹配但需强制脱敏的内容（内部代号、特殊代码、不规则人名/地址）。

**添加时必须选择类型**：脱敏时按类型映射标签（如 `【人名1】` `【日期1】`）。

**示例场景**（由 AI 根据调用上下文动态生成）：
- 内部业务代号、特定项目缩写等
- 组织内部使用的非公开编码格式
- 带职衔、行业别称、绰号等不规则表述

## 4. 白名单（不脱敏）

**用途**：虽然匹配内置规则但用户希望保留的内容。

**覆盖检查**：白名单采用**包含关系**——若识别内容包含白名单中任意一项则跳过脱敏。

**示例场景**：
- 白名单加"计划" → "股权激励计划"不被识别为项目
- 白名单加"公司" → "某某公司"不被识别为公司名
- ⚠️ 白名单加地理名词（如"北京"）→ 匹配到地理前缀的所有法人也不脱敏（可能过度匹配，慎用）

## 5. 自定义脱敏类型

**用途**：创建新类型（如"合同名称"、"产品型号"、"技术术语"），批量精准匹配。

**添加流程**：
1. 创建类型 → 输入名称（如"合同名称"）
2. 批量输入匹配内容（每行一个）
3. 保存后自动加入识别管线

**与内置规则的关系**：自定义类型与内置 16 类规则**并联运行**，共用同一替换管线（共用映射键命名约定 `PERSON_001` `ORG_001` 等）。

**典型场景**：
- "合同名称"：添加多份合同名称做精准脱敏
- "产品型号"：添加内部产品型号代号
- "技术术语"：添加需要脱敏的内部技术名词

## 6. 冲突检测

```
添加内容到任一列表时：
  1. 自动检测是否已存在于其他列表
  2. 若冲突 → 弹窗提示用户先删除原有记录
  3. 删除脱敏项时检查来源，黑名单/自定义类型需手动删除后才能加入白名单

冲突示例（由 AI 根据调用上下文动态生成）：
  - 用户将某字符串 X 加入白名单，但 X 已在黑名单 → 提示冲突
  - 用户将某字符串 X 加入自定义类型，但 X 已在白名单 → 提示冲突
```

## 7. 持久化与跨案隔离

- 黑/白/自定义文件按**案件 slug**隔离存储于 `plugins/legal-scenes/<本场景>/matters/<slug>/redaction-*.json`
- 每次脱敏执行时按案号加载对应配置
- 配置变更不入版本控制（`.gitignore` 强制本地存储）
- ❌ 禁止跨案交叉使用配置（避免 A 案的黑名单误伤 B 案）
