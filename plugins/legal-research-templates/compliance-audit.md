# Template: 合规审查

## 一、触发条件

律师输入包含：
- "合规审查" / "合规检查" / "合规审计"
- "这个合规吗" / "有没有合规风险"
- "内部合规" / "数据合规"

适用场景：内部合规审计、产品数据合规审查、内部规章合规性检查。

## 二、工作流（4 步）

### 步骤 1：要素提取
**调 atomic**：`legal-element-extraction`
- **输入**：审查对象（产品 / 业务 / 文档）+ 律师关注点
- **输出**：结构化要素

### 步骤 2：法条效力核查
**调 atomic**：`legal-norm-validity-check`
- **输入**：步骤 1 的相关法规
- **输出**：现行有效 / 已废止 / 已修订 状态

### 步骤 3：风险评估
**调 atomic**：`legal-risk-assessment`
- **输入**：步骤 1 + 2
- **输出**：风险等级（高/中/低）+ 关键风险点

### 步骤 4：冲突解决
**调 atomic**：`conflict-resolution`
- **输入**：步骤 3 的风险点
- **输出**：合规建议 + 整改路径

## 三、输入契约

律师需提供：
- **必需**：
  - 审查对象（产品 / 业务 / 文档）
  - 关注法规领域（如数据保护 / 劳动 / 反垄断）
- **可选**：
  - 内部已有规章
  - 历史审计发现
  - 监管最新动态

## 四、输出契约

```markdown
## 合规审查报告

### 一、审查对象要素
[结构化要素]

### 二、相关法规状态
[法规列表 + 现行有效状态 + 关键条款]

### 三、风险评估
[风险点 + 等级]

### 四、合规建议
- 立即整改：[高风险项]
- 计划整改：[中风险项]
- 持续监控：[低风险项]
```

## 五、失败 fallback

| 失败位置 | fallback 动作 |
|---|---|
| 步骤 2（法条核查）失败 | 跳过效力核查，标注"未核实现行性" |
| 步骤 3（风险评估）失败 | 输出定性分析（不评分） |
| 步骤 4（冲突解决）失败 | 输出风险点 + 建议专业咨询 |
| 整链失败 | 输出 [WARN] + 强烈建议律师人工审查 |

## 六、特别提示

合规是高风险决策。涉及：
- 个人信息（PIPL）
- 反垄断
- 跨境数据传输
- 金融业务
- 上市公司治理

agent 输出**仅供初步参考**，正式合规意见**必须由专业律师审核**。
